0먼저: 설명 대본 (외워두면 끝)

🗣️ 30초 버전 (엘리베이터)

"해충 상황을 한국어 문장으로 쓰면, 작은 AI 모델 두 개가 ① 무슨 해충인지와 ② 지금 뭘 해야 하는지를 판단하고, 거기에 ③ 어느 지역인지를 붙여서 관할 보건소 연락처 + 대응 가이드를 돌려줘요. 모델은 일부러 아주 작게(70KB) 만들어서 인터넷 없이 폰에서도 돌아가게 했어요."

🗣️ 3분 버전 (구조까지)

"핵심은 2개의 TFLite 텍스트 분류 모델이에요. 첫 번째는 문장 → 해충 7종, 두 번째는 문장 → 필요한 행동 4종(긴급·신고·안내·무관). 둘 다 bag-of-words라고, 문장의 단어들을 숫자 좌표로 바꿔 평균낸 뒤 분류하는 구조라 아주 가볍고 빨라요(수 밀리초, 오프라인).

그런데 작은 모델이라 함정 문장에 약해서, 모델 위에 규칙 레이어를 얹어 정확도를 끌어올려요. 그리고 두 모델의 판단이 어긋나는 '이음새'를 advisory.py 한 곳에서 보정하죠. 마지막으로 문장에서 지역(시군구)을 뽑아 관할 보건소를 붙이는데, 보건소는 거리가 아니라 행정구역 이름으로 매칭해요. 약점도 분명해요 — 평가 점수 99.8%는 학습과 같은 분포라 부풀려진 값이고, 진짜 자연어(처음 보는 표현)에선 더 낮아요. 그게 다음 과제예요."

1한 문장으로 이게 뭐냐

사람이 쓴 해충 제보 문장 한 줄을 받아서, 구조화된 방역 추천(무슨 해충·어떤 행동·어디 연락·대응법)으로 바꿔주는 경량 에이전트.

왜 만들었나 (북극성 연결): 대회 주제가 "제한된 자원에서 안정 동작하는 의사결정 AI"다. 그래서 "정확도만 높은 큰 모델"이 아니라 작고·빠르고·오프라인인 모델을 일부러 골랐다. 이 선택 하나가 모든 설계를 지배한다 — 기억할 핵심.

2큰 그림 — 문장 한 줄이 답이 되기까지

입력 문장→ ① 무슨 해충?
pest 모델→ ② 어떤 행동?
action 모델→ 이음새 보정
advisory→ ③ 어느 지역?
region→ ④ 보건소+가이드
recommend→ 응답

예: "동국대학교에 말벌집이 너무 많이 생겼습니다" → ① 말벌(wasp) · ② 긴급(emergency) · ③ 서울 중구 · ④ 서울중구보건소 ☎ 02-3396-6302 + 응급 가이드.

단계하는 일담당 파일
① pest문장 → 해충 7종 분류predict.py + tflite
② action문장 → 행동 4종 분류action_predict.py + tflite
이음새두 모델 판단 충돌 보정advisory.py
③ region문장 → 시군구region_resolve.py / region_detect.py
④ recommend해충+행동+지역 → 보건소+가이드recommend.py + offices_db.py

3"모델"이 도대체 뭐냐

여기서 "모델"은 문장을 넣으면 분류 결과를 뱉는 작은 수학 함수 덩어리다. 우리가 쓰는 형태는 bag-of-words(단어 봉지) 분류기.

비유: 문장을 단어들로 잘라 봉지에 다 넣고 흔든다(어순은 버린다). 그리고 "이 봉지 안에 어떤 단어들이 들었나"의 평균 느낌으로 판단한다. "말벌"이라는 단어가 들었으면 말벌 쪽으로 기운다. 순서를 안 본다는 게 핵심 — 가볍지만 한계의 뿌리이기도 하다(8번에서 다시 나옴).

TFLite = 그 함수를 작은 파일로 압축한 것

학습이 끝난 모델을 TFLite라는 포맷(.tflite)으로 변환하면 70KB짜리 파일 하나가 된다. 이걸 동봉하면 인터넷 없이 어디서나 즉시 로드해서 추론(수 밀리초)한다.

왜 작게? 큰 LLM은 정확하지만 무겁고 느리고 보통 인터넷이 필요하다. 대회·실사용 제약(오프라인·저사양)에선 못 쓴다. 70KB·CPU 수 ms·오프라인 = "제한된 자원에서 안정 동작"이라는 주제를 그대로 만족. 정확도를 조금 양보하고 크기·속도·오프라인을 산 것.

4전처리 — 왜 kiwi로 문장을 자르나

모델은 글자가 아니라 단어(토큰)를 본다. 그런데 한국어는 "말벌집이", "말벌집을", "말벌집은"처럼 같은 말에 조사·어미가 덕지덕지 붙는다. 이걸 그대로 다른 단어로 취급하면 모델이 헷갈린다.

kiwipiepy(한국어 형태소 분석기)로 문장을 형태소로 쪼개고 내용어만(명사·동사 등) 남긴다. "말벌집이 생겼어요" → 말벌 집 생기다 식.

절대 규칙(어기면 조용히 망가짐): 학습할 때와 추론할 때 토큰화가 완전히 똑같아야 한다. 그래서 토큰화 정의(tokenize())는 predict.py에 딱 하나만 두고 학습 스크립트가 그걸 import해서 쓴다. 두 곳이 어긋나면 모델은 본 적 없는 토큰을 받아 엉뚱한 답을 낸다. (학습 단어장은 vocab.json으로 저장해 추론 때 그대로 복원.)

5모델 구조 한 줄씩 뜯어보기

kiwi 토큰화(내용어만)
  → TextVectorization(SEQ_LEN=32)      # 단어 → 정수 번호, 길이 32로 맞춤
  → Embedding(2000, 32, mask_zero)     # 각 단어 → 32차원 좌표
  → GlobalAveragePooling1D()           # 단어 좌표들을 "평균" → 문장 한 점
  → Dense(32, relu)                    # 한 번 비선형 변환
  → Dense(클래스수, softmax)            # 점수 → 확률(합 100%)
  → TFLite 변환
줄쉽게 말하면
TextVectorization단어를 사전 번호로. "말벌"=17번 식. 길이를 32로 통일(짧으면 0으로 채움).
Embedding각 단어 번호를 32개 숫자 좌표로. 비슷한 뜻의 단어가 가까이 모이도록 학습된다.
GlobalAveragePooling문장 속 모든 단어 좌표를 평균내 문장을 점 하나로 요약. ← 여기가 가벼움의 비결이자 약점의 원천.
Dense + softmax그 점을 보고 클래스별 점수를 내고, 합이 100%가 되는 확률로 바꾼다. 가장 높은 게 답.
softmax 비유: 시험 점수(말벌 8점, 모기 1점, 없음 1점)를 "말벌 80% · 모기 10% · 없음 10%"처럼 비율로 바꾸는 것. 이 비율이 "확신도".

6왜 모델을 둘로 나눴나

모델입력 → 출력클래스
pest 분류기문장 → 무슨 해충mosquito·cockroach·lovebug·wasp·tick·bedbug·none (7)
action 분류기문장 → 필요한 행동emergency·dispatch·guide·none (4)

"무슨 해충"과 "뭘 해야 하나"는 다른 질문이다. 같은 말벌이라도 "말벌집 생겼어요"(긴급)와 "말벌 안 생기게 하려면?"(안내)은 행동이 다르다. 그래서 책임을 둘로 분리해 각자 작은 모델이 한 가지만 잘하게 했다.

action의미연결
emergency쏘임·호흡곤란·말벌집119 + 관할 보건소
dispatch대량 출몰·"방역 필요"·신고관할 보건소
guide"어떻게 없애요?" 예방·정보 문의대응 가이드
none해충 무관(잡담·애정표현)—

7모델만으론 안 된다 — 규칙 레이어

작은 평균 모델은 함정 문맥에 약하다. 그래서 모델 확률 위에 규칙 3종을 얹는다(정확도 대부분이 사실 여기서 나온다 — predict.py).

규칙하는 일
normalize_text()표기 통일. "러브 버그/love bug/말 벌" → 학습 표기로. 단 영어 love bug는 제보 문맥일 때만 러브버그(I love bugs는 무관).
calibrate_probs()명확한 해충명 + 제보 문맥어가 있으면 확신도 바닥(0.85) 보장. 단 비제보 문맥이 섞이면 안 함.
apply_context_overrides()"차량 바퀴에 벌레가 꼈다"를 바퀴벌레로 오인하지 않게 none으로 강제.
설명 포인트: "AI 모델 하나로 끝"이 아니라 작은 모델 + 사람이 짠 규칙의 협업이다. 모델은 대충 잡고, 규칙이 함정을 막는다. 이게 '경량'을 유지하면서 정확도를 사는 방법.

8두 모델의 이음새 — 우리가 고친 버그 (★ 설명 황금소재)

pest와 action은 독립된 두 모델이라 판단이 어긋날 때가 있다. 그 틈을 advisory.correct_seam() 한 곳에서 메운다(두 앱이 공유).

① pest=none 인데 action≠none  → 원문에 해충 단어 있으면 그 해충으로 복구
② pest≠none 인데 action=none  → none 대신 dispatch(신고)로 강등 방지
실제 겪은 버그(완벽한 사례): "말벌집이 생겼어요"는 wasp(100%)인데, "말벌집이 너무 많이 생겼습니다"는 none(88%)로 샜다.
왜? 5번의 GlobalAveragePooling(평균) 때문. 강한 단어 "말벌"에 중립 단어 "너무·많이"와 격식체 "생겼습니다"가 더해지자 평균이 흐려져 none 쪽으로 넘어간 것. ← 구조의 약점이 그대로 드러난 순간.
고친 방법: action 모델은 이 문장을 emergency로 맞혔으므로, "pest만 none이고 원문에 '말벌'이 있으면 wasp로 복구"하도록 ①을 일반화. (예전엔 질문형에만 적용돼 이 케이스가 "해충 무관"까지 새던 사각지대였다.)
운영 교훈도 같이: 코드를 고쳐도 이미 떠 있는 서버는 옛 코드를 물고 있어 그대로 오답을 냈다(Flask가 자동 리로드 안 함). "고쳤다"는 새 프로세스 테스트가 아니라 실제 도는 서버를 재시작해 확인한 뒤에만.

9어디인지 알아내기 — 지역 해소

문장에서 장소를 뽑아 시군구로 바꾼다. 순서가 핵심이고, 키 유무로 갈린다(region_resolve.py):

순서방법오프라인?
1. 룰정규 시군구 직접 매칭 + 별칭(광화문→종로구, 해운대→해운대구)✅
2. 캐시전에 해소한 장소면 geocode.json에서 바로✅
3. 라이브처음 보는 장소 → Kakao 검색 → 좌표 → 행정구역. 성공하면 캐시에 적재키+망 필요(1회)
반직관 함정(꼭 기억): 이름에 지역이 박혀도 좌표가 진실이다. 예) "서울대공원"은 이름은 서울이지만 실제 위치는 과천시. 룰/이름으로 풀면 틀리고, 좌표로 풀면 맞다. 그래서 라이브 경로가 룰보다 정확할 수 있다.
거친 시 정밀화: "단국대 천안캠퍼스"의 "천안"만 잡으면 천안시(구 없음)인데, 보건소는 천안시 동남구/서북구처럼 구 단위라 관할을 못 짚는다. 그래서 구를 가진 시는 좌표로 구까지 정밀화한다.

10어디에 연락? — 추천 + 보건소

recommend()가 해충 정보(pest_info.json) + 관할 보건소(offices_db)를 합쳐 완성된 답변 텍스트를 만든다. 119는 긴급 공통.

핵심: 보건소는 가장 가까운 곳이 아니라 행정구역 이름으로 매칭한다. 옆 구의 더 가까운 보건소를 잡으면 관할이 틀린다. "좌표 → 시군구 → 이름 매칭"이 정석. 시군구를 못 잡고 시도만 있으면 시도 대표 보건소로 강등(아무것도 못 주는 것보단 낫게). 전화번호는 공공데이터 원본만 쓴다(추측 0).

데이터: 보건복지부 전국 지역보건의료기관(2022) → 224개 시군구 / 259곳 보건소.

12솔직한 약점 — 이걸 같이 말해야 진짜 이해

① 99.8%는 부풀려진 값. 평가셋이 학습과 같은 증강 템플릿 분포(in-distribution)라 높게 나온다. 처음 보는 자연스러운 표현(OOD)에선 떨어진다 — 이음새 정확도 ≈ 88%.
② pest 모델 희석 약점(8번). "말벌집이 너무 많이 생겼습니다" 같은 표현을 none으로 흘린다. 챗은 advisory가 복구하지만, 분석앱(8600)은 모델 원본 확신도를 화면에 노출해서 라벨만 못 바꾼다(89% 해충없음 + 말벌 카드는 모순). 진짜 해결은 모델/재학습 레벨.
③ 보건소 번호 vintage. 2022년 기준이라 일부 번호가 현행과 다를 수 있다 → 고빈도 보건소는 공식 페이지로 spot-check.
④ 오프라인은 어디까지? 모델 추론은 100% 오프라인(tflite·kiwi·룰·보건소 룩업 전부 망 0). 단 하나의 온라인 = 캐시·룰에 없는 새 장소를 좌표로 해소할 때만 Kakao/Nominatim 호출, 1회 후 캐시. 망 없으면 아는 장소만 풀고 모르는 장소는 "지역 알려주세요"로 폴백 — 해충·행동 판정은 그대로.

13누가 물어도 답하는 Q&A

Q. 그냥 ChatGPT 쓰면 되지 왜 직접 모델을?

A. 대회·실사용 제약이 오프라인·저사양·고속이라 큰 LLM을 못 쓴다. 70KB 모델은 인터넷 없이 폰에서도 수 ms에 돈다. "제한된 자원에서 안정 동작"이 주제 그 자체.

Q. bag-of-words면 어순을 무시하는데 괜찮아?

A. 짧은 제보 문장 분류엔 충분히 통한다(가볍다). 대신 어순/문맥을 못 봐서 생기는 약점은 규칙 레이어와 이음새 보정으로 메운다. 그래도 남는 한계가 OOD(8·12번)다 — 숨기지 않는다.

Q. 모델이 틀리면?

A. 세 겹의 안전망: ① 규칙 레이어가 함정 문맥 차단, ② 두 모델 교차검증(이음새 보정), ③ 지역 못 잡으면 시도 보건소로 강등·번호는 원본만. 그래도 99.8%는 과장이라고 먼저 밝힌다.

Q. "완전 오프라인"이라며 Kakao는 왜?

A. 모델은 완전 오프라인. Kakao는 처음 보는 장소 이름을 좌표로 바꿀 때만, 그것도 1회(이후 캐시). 모델 판정과는 별개의 부가기능이고 망 없으면 우아하게 폴백한다.

Q. 학습 데이터는?

A. 손으로 만든 까다로운 175개 시드 문장을 템플릿으로 증강해 ≈5,314문장. 평가셋과 학습셋 교집합이 0임을 assert로 강제(누수 방지). 테스트 프레임워크 대신 eval_*.py가 게이트 역할.

Q. 제일 자랑할 점 하나만?

A. 정직한 경량 설계. 70KB 모델 + 규칙 + 교차검증으로 오프라인에서 도는데, 점수를 부풀리지 않고 약점(OOD)까지 측정해 다음 과제로 못박았다.

이 리포트는 pest-sns 코드(predict.py·advisory.py·region_resolve.py·recommend.py·offices_db.py 등)와 검증 결과를 바탕으로 작성됨. 숫자·동작이 바뀌면 갱신.
